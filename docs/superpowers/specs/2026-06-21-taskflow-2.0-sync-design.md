# TaskFlow 2.0 端到端加密同步架构设计

> 状态：设计完成，待实现 MVP
> 关联文档：[生物识别解锁设计](2026-06-21-biometric-unlock-design.md)

## 1. 目标

- 在多台设备之间安全同步 TaskFlow 数据（任务、保险库、分类）。
- 同步服务器/中继**无法读取**用户数据，实现端到端加密（E2EE）。
- 支持离线操作，联网后自动同步。
- 新设备加入需经现有设备授权，防止未授权访问。

## 2. 同步拓扑

采用**混合模式**：

1. **局域网直连（首选）**：设备在同一 LAN 下通过 mDNS 发现，直接建立 TCP/WebSocket 连接交换同步数据。
2. **自托管中继（fallback）**：无法直连时，通过可选的 Node.js 中继服务器交换加密同步包。
3. **离线优先**：所有写操作先落地本地 SQLite，同步线程在后台择机推送/拉取。

```
设备 A  ◄────LAN 直连────►  设备 B
  │
  └──── 加密同步包 ────►  自托管中继 Server  ◄──── 加密同步包 ────  设备 C
```

## 3. 密钥体系

### 3.1 同步主密钥（Sync Master Key, SMK）

- 256-bit 随机密钥，与登录主密码解耦。
- 用于加密所有同步记录（AES-256-GCM）。
- 每台设备使用 Electron `safeStorage` 保存 SMK。

### 3.2 设备身份密钥

- 每台设备生成 Ed25519 密钥对作为设备身份。
- 公钥在首次配对时通过加密通道广播给其他设备。
- 私钥保存在 `safeStorage` 中。

### 3.3 会话密钥

- 两台设备通信时，使用 ECDH (X25519) 临时密钥对协商会话密钥。
- 会话密钥用于加密传输层数据，生命周期限定在一次同步会话内。

### 3.4 新设备加入流程

1. 新设备生成临时 ECDH 密钥对。
2. 现有设备展示**配对码**（8 位数字 / Base32）或**二维码**。
3. 新设备输入配对码，双方通过中继或 LAN 完成 ECDH 握手。
4. 握手成功后，现有设备将 SMK 加密发送给新设备。
5. 新设备保存 SMK，后续同步使用该密钥。

## 4. 数据模型

### 4.1 同步记录（sync_records）

```sql
CREATE TABLE IF NOT EXISTS sync_records (
  id TEXT PRIMARY KEY,              -- table_name:record_id:v:version
  table_name TEXT NOT NULL,         -- tasks / vault_items / categories
  record_id TEXT NOT NULL,          -- 原始记录 ID
  version INTEGER NOT NULL,         -- 乐观锁版本
  encrypted_payload BLOB NOT NULL,  -- AES-256-GCM 加密后的 JSON
  updated_at INTEGER NOT NULL,      -- 毫秒时间戳
  deleted INTEGER DEFAULT 0         -- 软删除标记
);
```

### 4.2 同步状态（sync_state）

```sql
CREATE TABLE IF NOT EXISTS sync_state (
  id INTEGER PRIMARY KEY CHECK (id = 1),
  local_clock INTEGER NOT NULL DEFAULT 0,   -- 本地逻辑时钟
  last_sync_at INTEGER                        -- 上次成功同步时间
);
```

### 4.3 设备注册表（sync_devices）

```sql
CREATE TABLE IF NOT EXISTS sync_devices (
  device_id TEXT PRIMARY KEY,       -- 设备公钥指纹
  public_key TEXT NOT NULL,         -- Base64 Ed25519 公钥
  name TEXT,                        -- 设备别名
  paired_at INTEGER NOT NULL,
  last_seen_at INTEGER
);
```

## 5. 加密细节

### 5.1 记录加密

```
encryptRecord(plaintextJSON, smk):
  iv     = randomBytes(12)
  cipher = AES-256-GCM(smk, iv)
  ciphertext = iv || authTag || encrypted
```

- SMK 不直接用于传输层，仅用于加密持久化同步记录。
- 同一记录的多个版本使用同一 SMK，不同 IV。

### 5.2 传输加密

- ECDH 握手后得到共享密钥 `sharedSecret`。
- 使用 `sharedSecret` 派生会话密钥：`HKDF-SHA256(sharedSecret, salt, 'taskflow-sync-v1')`。
- 会话消息使用 AES-256-GCM 加密。

## 6. 同步协议

### 6.1 消息类型

```typescript
type SyncMessage =
  | { type: 'HELLO'; deviceId: string; publicKey: string; nonce: string }
  | { type: 'OFFER'; encryptedPayload: string }        // ECDH 公钥 + 签名
  | { type: 'ANSWER'; encryptedPayload: string }
  | { type: 'MANIFEST'; records: { id: string; updatedAt: number; hash: string }[] }
  | { type: 'REQUEST'; recordIds: string[] }
  | { type: 'BATCH'; records: SyncRecord[] }
  | { type: 'ACK'; receivedIds: string[] }
  | { type: 'ERROR'; code: string; message: string };
```

### 6.2 同步流程

1. **发现**：LAN 中通过 mDNS 发现同账号设备；或连接配置的中继服务器。
2. **握手**：ECDH + Ed25519 签名验证，确认双方身份。
3. **交换清单**：双方发送各自拥有的记录摘要（id + updatedAt + hash）。
4. **请求缺失**：向对端请求本地缺失或较旧的记录。
5. **批量同步**：对端发送 `BATCH` 消息，本地解密并写入。
6. **确认**：发送 `ACK`，更新 `last_sync_at`。

### 6.3 冲突解决（MVP）

- 采用 **last-write-wins**：比较 `updated_at`，保留时间戳更大的版本。
- 版本字段 `version` 用于检测并发写入；如双方版本同时增加，后续可升级为 CRDT 或手动合并。

## 7. 中继服务器

### 7.1 职责

- 为无法直连的设备转发加密消息。
- 临时存储加密同步包（TTL 7 天）。
- 不保存也不具备解密 SMK 的能力。

### 7.2 接口（WebSocket）

```
POST /register-device
POST /claim-pairing-code
WS /sync?deviceId=...
```

### 7.3 部署

- 默认不启用；用户在设置中填入中继地址后开启。
- 提供 Docker 镜像：`taskflow/sync-relay:latest`。

## 11. 安全与隐私

- 中继服务器仅存储加密 blob，数据库索引按 `device_id` + `message_id` 分片。
- 配对码限时有效（5 分钟），过期自动失效。
- 设备可随时在设置中「移除设备」，移除后该设备无法再参与同步。
- 所有传输层消息包含 Ed25519 签名，防止中间人篡改。

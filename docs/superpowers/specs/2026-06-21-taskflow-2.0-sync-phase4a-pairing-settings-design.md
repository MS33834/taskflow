## 1. 目标

让桌面端用户能够：
1. 在设置页开启/关闭同步、配置自托管中继地址。
2. 查看已配对设备列表，移除设备。
3. 作为旧设备生成 8 位配对码（或二维码），供新设备加入。
4. 作为新设备输入配对码，完成与旧设备的 ECDH 握手并安全接收 SMK。

Phase 4a 不实现后台自动同步调度器（Phase 4b），也不实现安全硬化（Phase 4c）。

## 2. 范围

### 包含
- 桌面端主进程配对服务（`pairingService.ts`）
- 中继 HTTP 客户端封装（`relayClient.ts`）
- IPC 通道：`syncChannels.ts` / preload / renderer API
- 渲染层状态：`syncStore.ts`
- 设置页同步模块：`SyncSettingsPanel.tsx`、`PairingDialog.tsx`、`DeviceList.tsx`
- 单元测试与集成测试

### 不包含
- 后台同步调度、LAN/Relay 自动选择（Phase 4b）
- 传输层重放保护、握手超时、设备吊销贯通（Phase 4c）
- 移动端同步
- 官方托管中继

## 4. 关键组件

### 4.1 relayClient.ts
封装对 relay 的 HTTP 调用：
- `registerDevice(identity)` → `{ token, wsUrl }`
- `createPairingCode(token, identity)` → `{ code, expiresAt }`
- `claimPairingCode(code, identity)` → `{ token, wsUrl }`
- `refreshToken(token)` → `{ token }`

所有请求携带 Ed25519 签名，与 relay `routes.ts` 的校验对齐。

### 4.2 pairingService.ts
负责完整配对流程：
- **旧设备侧**：
  1. 注册到 relay（若未注册）。
  2. 调用 `/pairing-codes` 生成 8 位数字码。
  3. 启动一个 responder `RelayTransport` 等待新设备连接。
  4. 握手成功后，通过新增 `SMK_TRANSFER` 消息将 SMK 加密发送给新设备。
  5. 将新设备公钥写入 `sync_devices`。
- **新设备侧**：
  1. 输入配对码。
  2. 调用 `/claim-pairing-code` 获得 token。
  3. 通过 `RelayTransport` 连接旧设备（initiator）。
  4. 完成 `SyncSession` 握手。
  5. 接收 `SMK_TRANSFER`，保存 SMK 到 `safeStorage`。
  6. 将旧设备公钥写入 `sync_devices`。

### 4.3 syncChannels.ts
暴露给渲染层的 IPC：
- `sync:getState` → 返回 `{ enabled, relayUrl, devices, lastSyncAt }`
- `sync:setEnabled(enabled)`
- `sync:setRelayUrl(url)`
- `sync:generatePairingCode` → 返回 `{ code, expiresAt }`
- `sync:claimPairingCode(code)` → 返回 `{ success, deviceId? }`
- `sync:removeDevice(deviceId)`
- `sync:onStateChanged` (push)

### 4.4 syncStore.ts
Zustand store 管理：
- `enabled`、`relayUrl`
- `devices` 列表（设备 ID、名称、上次在线时间）
- `pairingCode` / `pairingExpiry`
- `loading` / `error`

### 4.5 UI 组件
- `SyncSettingsPanel`：开关同步、中继地址输入、设备列表容器。
- `RelayUrlInput`：验证 URL 格式，支持 `ws://` / `wss://`。
- `DeviceList`：展示已配对设备、移除按钮、上次同步时间。
- `PairingDialog`：旧设备展示配对码与二维码；新设备输入配对码。

## 5. 数据流

### 旧设备生成配对码
```
用户点击「添加设备」
  └─► pairingService.registerAndCreatePairingCode()
      ├─► relayClient.registerDevice(identity)
      ├─► 保存 token 到本地安全存储
      ├─► relayClient.createPairingCode(token, identity)
      └─► 返回 code → 渲染层展示
```

### 新设备输入配对码
```
用户输入配对码
  └─► pairingService.claimPairingCodeAndPair(code)
      ├─► relayClient.claimPairingCode(code, identity)
      ├─► 保存 token
      ├─► 创建 RelayTransport 连接旧设备
      ├─► SyncSession 握手
      ├─► 接收 SMK_TRANSFER 并保存 SMK
      └─► 注册旧设备到 sync_devices
```

## 6. 安全与隐私

- **配对码**：8 位数字，5 分钟 TTL，一次性，最多 5 次尝试，每设备 10 秒速率限制（由 relay 强制执行）。
- **SMK 传输**：仅通过已验证的 `SyncSession` 发送，使用临时会话密钥加密；relay 不持有 SMK。
- **Token 存储**：relay auth token 使用 `safeStorage` 加密保存。
- **设备移除**：Phase 4a 中仅删除本地 `sync_devices` 记录与 token；即时吊销会话留到 Phase 4c。
- **URL 验证**：中继地址必须显式指定协议，禁止明文提示用户。

## 7. 测试策略

### 单元测试
- `pairingService.test.ts`：mock relayClient 与 SyncSession，验证流程调用顺序。
- `relayClient.test.ts`：mock `fetch`/`https`，验证请求体与签名构造。
- `SyncSettingsPanel.test.tsx`：React Testing Library，验证开关、按钮状态。

### 集成测试
- `pairing.integration.test.ts`：启动真实 relay，两个内存数据库实例完成旧设备→新设备配对，验证 SMK 一致、设备注册成功。

### 验收标准
- [ ] 设置页出现「同步」模块。
- [ ] 开启同步后可配置中继地址，关闭后清除相关状态。
- [ ] 旧设备可生成 8 位配对码，5 分钟内过期。
- [ ] 新设备输入正确配对码后，可与旧设备完成握手并接收 SMK。
- [ ] 配对成功后双方 `sync_devices` 均包含对方。
- [ ] 桌面端 `npm run typecheck && npm run lint && npm run test` 全部通过。
- [ ] relay 端 `npm run typecheck && npm run test` 全部通过。

## 8. 待决策事项

1. 是否允许同一台设备同时作为旧设备和新设备？（建议禁止）
2. 配对码是否必须展示二维码？（MVP 建议先 8 位数字，二维码为可选增强）
3. 新设备首次启用同步时，是否自动生成 SMK 还是先配对再生成？（建议先配对再生成，旧设备始终生成/持有 SMK）
4. 设备别名是否允许用户编辑？（MVP 使用默认名如「Device-ABCD」）
